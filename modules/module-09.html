Bridge from M08

In M08 you learned QAOA a gate-model algorithm that encodes optimisation problems as Hamiltonians and uses variational circuits. Quantum annealing is a completely different physical approach to the same class of problems. Instead of gate circuits, it uses a slow evolution of a physical quantum system from a known ground state to the ground state of the problem Hamiltonian. D-Wave's Advantage processor with 5,627 qubits is the only commercially available quantum annealer. Understanding when to use annealing versus QAOA is a key advisory decision and your platform already makes it.

01 Annealing versus gate-model two very different machines

Quantum annealing and gate-model quantum computing (IBM, IonQ, Google) are fundamentally different technologies. They share the word "quantum" and both tackle optimisation, but they work differently, have different strengths, and are not interchangeable.

PropertyQuantum Annealing (D-Wave)Gate-model (IBM/IonQ)
Physical mechanismAdiabatic evolution of physical spinsDiscrete unitary gate operations
Qubit count (2024)5,627 (Advantage) connected127–1,000+ (noisy, limited connectivity)
Problem typeIsing/QUBO only binary optimisationAny quantum algorithm (QAOA, VQE, Grover, Shor)
Programming modelSpecify the QUBO matrix Q, submit, get sampleBuild a circuit gate by gate
No circuit depth limitYes annealing time, not circuit depthNo coherence time limits depth
UniversalityNo only Ising/QUBO problemsYes universal quantum computer
Classical benchmarkSimulated annealing, tabu searchOR-Tools, Gurobi (for optimisation)
💡 Why D-Wave has so many more qubits

Gate-model qubits require extreme isolation and precise microwave control scaling is hard. D-Wave's qubits are superconducting flux qubits that are physically coupled in a fixed topology (Pegasus). They do not need to run arbitrary circuits only Ising evolution. This specialisation allows D-Wave to build processors with thousands of qubits that no gate-model machine can yet match. The tradeoff is that you can only solve Ising/QUBO problems, not arbitrary quantum algorithms.

02 How quantum annealing works the physical intuition

Classical simulated annealing is a familiar algorithm: start with a random solution, make random moves, accept bad moves with decreasing probability as a "temperature" drops. Quantum annealing replaces thermal fluctuations with quantum tunnelling.

🔮 The landscape analogy

Imagine the solution space as a hilly landscape. Each valley is a local minimum a good-but-not-optimal solution. The global minimum is the optimal solution. Classical algorithms roll downhill and get stuck in local valleys. Quantum tunnelling lets the system pass through the hills rather than over them, potentially finding the global minimum faster. The catch: tunnelling works best for thin barriers. Wide flat barriers (common in large problems) are not tunnelled through efficiently the system still gets stuck.

1
Initialise in the ground state of a transverse field
Start with a strong transverse magnetic field Hₜ applied to all qubits. The ground state of this Hamiltonian is easy to prepare: every qubit is in equal superposition |+⟩ = (|0⟩+|1⟩)/√2. This represents all possible solutions simultaneously.
2
Slowly evolve toward the problem Hamiltonian
Over the anneal time (typically 20–2000 microseconds), slowly reduce the transverse field and increase the problem Hamiltonian H(s) = (1−s)Hₜ + s·Hproblem where s goes from 0 to 1. By the adiabatic theorem, if this evolution is slow enough, the system stays in the ground state throughout ending in the ground state of Hproblem, which is the optimal solution.
3
Read out the final spin configuration
At the end of the anneal, measure all qubits. Each qubit is now in state |0⟩ or |1⟩. The resulting bit string is a candidate solution. Run many anneals (typically 1,000–10,000) and take the lowest-energy result as the answer. The repetition is important: annealing is probabilistic and does not always find the optimal solution.
⚠ The adiabatic theorem requires slow evolution

The adiabatic theorem guarantees the ground state is maintained only if the anneal is infinitely slow. In practice, the anneal time is finite and the system may undergo Landau-Zener transitions at the minimum energy gap jumping out of the ground state. The minimum gap (the hardest point of the problem) determines how slowly you must anneal. For hard instances, the minimum gap is exponentially small and the required anneal time is exponentially long no quantum speedup in the worst case.

03 The Pegasus topology why not all qubits connect to all others

D-Wave's Advantage processor uses the Pegasus topology: a specific graph where each qubit connects to at most 15 others. This is not fully connected on a 5,627-qubit Pegasus graph, most pairs of qubits have no direct coupling.

This matters because your QUBO problem may require couplings between any pair of variables. When two variables need to interact but their qubits are not directly connected in Pegasus, D-Wave's embedding algorithm creates a chain of connected qubits to represent one logical variable. Chains use multiple physical qubits to represent one logical variable.

Logical qubits vs physical qubits the embedding overhead
Logical qubits: The variables in your QUBO problem. For a 50-city TSP with one-hot encoding: 50×50 = 2,500 logical qubits.

Physical qubits: The actual qubits on the D-Wave chip used to represent your logical problem after embedding. Each logical qubit in a dense QUBO typically requires 3–8 physical qubits due to chain overhead. 2,500 logical → ~7,500–20,000 physical qubits. This exceeds Advantage's 5,627 physical qubits so a 50-city TSP does not fit on current D-Wave hardware without a smarter formulation.

Chain strength: The coupling strength between physical qubits in a chain. Too weak → chains break (qubits disagree on the logical value). Too strong → chains dominate the problem energy, masking the actual objective. Tuning chain strength is the most important parameter in D-Wave programming.

04 Building a QUBO matrix the complete recipe

Every problem you submit to D-Wave must be expressed as a QUBO. The process is always the same: binary variables for each decision, costs for what you want to optimise, and penalty terms for constraints that must be satisfied.

1
Identify binary variables
One binary variable xᵢ per decision. For job scheduling: xⱼₜ = 1 if job j is scheduled in slot t, 0 otherwise. For portfolio: xᵢ = 1 if asset i is included in the portfolio. These become your qubits.
2
Write the objective function
The thing you want to minimise. For portfolio: minimise risk − λ×return, where the covariance matrix σᵢⱼ gives couplings between assets: Q[i][j] += λ_risk × σᵢⱼ for off-diagonal, and Q[i][i] -= λ_return × rᵢ for diagonal returns. Minimising this QUBO selects the portfolio with maximum risk-adjusted return.
3
Add penalty terms for constraints
Constraints become penalty terms added to the QUBO. "Select exactly k assets" becomes: penalty × (Σᵢ xᵢ − k)². Expand this: penalty × (Σᵢ xᵢ² + 2Σᵢ﹤ⱼ xᵢxⱼ − 2k Σᵢ xᵢ + k²). The constant k² drops out. This adds penalty to Q[i][i] and 2×penalty to Q[i][j] for i≠j. The penalty coefficient must be large enough to make any constraint violation more costly than any feasible solution improvement.
4
Choose the penalty coefficient
The penalty λ must satisfy: λ > max possible improvement from violating the constraint. Too small → constraint violations in the solution. Too large → the penalty dominates the objective and the optimiser ignores the actual cost function, finding feasible but poor-quality solutions. Finding the right λ is problem-specific and often requires experimentation.

06 Three real QUBO formulations

Job Scheduling n jobs, T time slots, precedence constraints
Variables: xⱼₜ = 1 if job j is assigned to slot t. Total: n×T binary variables.

Objective: Minimise total weighted completion time: Σⱼ Σₜ t × wⱼ × xⱼₜ

Constraint 1 each job scheduled exactly once: Σₜ xⱼₜ = 1 for each j
Penalty: λ₁ × Σⱼ (Σₜ xⱼₜ − 1)²

Constraint 2 at most one job per slot: Σⱼ xⱼₜ ≤ 1 for each t
Penalty: λ₂ × Σₜ Σⱼ﹤ₖ xⱼₜ × xₖₜ

Constraint 3 precedence (job A before job B): Σₜ t × xₐₜ < Σₜ t × x_bₜ
This is harder to encode as QUBO requires slack variables or time-indexed formulation.
Portfolio Selection n assets, select k with maximum risk-adjusted return
Variables: xᵢ = 1 if asset i is included. Total: n binary variables.

Objective (Markowitz): Minimise λ_risk × xᵀΣx − λ_return × μᵀx
where Σᵢⱼ is the covariance matrix and μᵢ is expected return.

QUBO form: Q[i][j] = λ_risk × Σᵢⱼ for i≠j; Q[i][i] = λ_risk × Σᵢᵢ − λ_return × μᵢ

Constraint select exactly k assets: Penalty λ_c × (Σᵢ xᵢ − k)²
Add: λ_c × 2 to Q[i][j] for i≠j; −2λ_c × k to Q[i][i]

07 Chain breaks the most common failure mode

When you run a problem on D-Wave, the embedding maps logical variables to chains of physical qubits. At the end of the anneal, qubits in a chain should all agree they should all be 0 or all be 1. When they disagree, it is called a chain break.

⚠ Common mistake
✗  "D-Wave returned nonsense the solution violates all my constraints"
✓  Your chain strength λ_chain is too small. Chains broke during annealing, producing random logical values. Increase chain strength. Rule of thumb: chain_strength ≈ max(|Q[i][j]|) × 1.5 to 2.0
Chain break rateMeaningAction
< 5%Normal most chains are intactResults are likely valid
5–20%Moderate solution quality may be degradedIncrease chain strength by 1.5×
> 20%High most solutions are invalidDouble chain strength; consider Leap Hybrid instead
✓ Check your understanding

B~300 100 logical × 3 physical per chain
C5,627 always uses the whole chip
D10,000 QUBO variables × 100
✓ Correct. Each logical variable requires an average of 3 physical qubits due to embedding overhead in the Pegasus topology. 100 logical variables × 3 physical per chain = ~300 physical qubits. This fits comfortably within the 5,627-qubit Advantage processor. In contrast, a 1,000-asset portfolio would need ~3,000 physical qubits still within Advantage's capacity but with limited headroom.
✓ Check your understanding
Your D-Wave solution violates the "select exactly 5 assets" constraint it selected 7. What is the most likely cause?
AD-Wave hardware is broken
BThe penalty coefficient is too small constraint violation costs less than the return from 7 assets
CThe QUBO matrix Q was built incorrectly
DThe anneal time was too short
✓ Correct. Constraint violations in D-Wave results almost always mean the penalty coefficient λ_c is too small. The annealer found that violating the "select exactly 5" constraint (paying a small penalty) gives a better total energy than satisfying it. Solution: increase λ_c until it exceeds the maximum possible gain from violating the constraint. A good starting point: λ_c = 2 × max(|return values|).

Key Ocean SDK concepts
sampler.sample_qubo(Q, num_reads=1000)Run 1000 independent annealing runs. Each run returns one solution. The sampler returns all 1000 solutions ranked by energy. The best solution (lowest energy) is response.first. More reads = better chance of finding the global optimum.
best.energyThe QUBO objective value for this solution. Lower is better (QUBO minimises). A valid, optimal solution has energy = objective value + 0 penalty. Any positive penalty component in the energy means a constraint was violated.
best.sampleA dict mapping variable name → 0 or 1. This is the actual assignment which variables are set to 1 in this solution. Read it to extract your schedule, portfolio, or partition.

NB 01 · Penalty tuning
What happens when your penalty is too small
Deliberately set the penalty too small and observe constraint violations. Then find the correct penalty by comparing with brute-force enumeration. This is the most important calibration skill for D-Wave programming.

Define a reusable QUBO builder that takes the penalty coefficient as a parameter. Then test with penalty=1 too weak to enforce constraints to see violations, and then with penalty=10 strong enough to see all constraints satisfied.

[ ]:
def build_scheduling_qubo(jobs, slots, weights, penalty): """Build scheduling QUBO with given penalty coefficient.""" Q = {} for j in jobs: for t in slots: k = (var(j,t), var(j,t)) Q[k] = Q.get(k,0) + (t+1)*weights[j] + penalty*(1-2) for t1,t2 in itertools.combinations(slots,2): k = (var(j,t1),var(j,t2)) Q[k] = Q.get(k,0) + 2*penalty for t in slots: for j1,j2 in itertools.combinations(jobs,2): k = (var(j1,t),var(j2,t)) Q[k] = Q.get(k,0) + 2*penalty return Q # Test with penalty=1 (too small) print("=== PENALTY = 1 (too small) ===") Q_bad = build_scheduling_qubo(jobs, slots, weights, penalty=1) resp_bad = SimulatedAnnealingSampler().sample_qubo(Q_bad, num_reads=500) b = resp_bad.first print(f"Energy: {b.energy:.2f}") for j in jobs: slots_used = [t for t in slots if b.sample[var(j,t)]==1] ok = '✓' if len(slots_used)==1 else f'✗ VIOLATION (slots {slots_used})' print(f" Job {j}: {ok}") print("\n=== PENALTY = 10 (correct) ===") Q_good = build_scheduling_qubo(jobs, slots, weights, penalty=10) resp_good = SimulatedAnnealingSampler().sample_qubo(Q_good, num_reads=500) b2 = resp_good.first print(f"Energy: {b2.energy:.2f}") for j in jobs: slots_used = [t for t in slots if b2.sample[var(j,t)]==1] ok = '✓' if len(slots_used)==1 else f'✗ VIOLATION' print(f" Job {j}: slot {slots_used}: {ok}")

📌 Key insight

With penalty=1, the annealer assigned job A to two slots the lower energy from double-assigning (saving objective cost) outweighed the tiny penalty. With penalty=10, constraints are correctly enforced. The minimum valid penalty must exceed the maximum possible objective gain from any constraint violation. Always verify by checking all constraints on every solution.
