Hybrid Quantum-Classical Workflows

How real quantum applications combine classical and quantum computation. The four hybrid patterns, when each applies, how to partition a problem, and what production hybrid architecture looks like in code and in practice.

01 Why everything quantum is already hybrid

There is no purely quantum application today. Every useful quantum computation involves a classical computer doing significant work before, during, and after the quantum part. Understanding this partnership is more important than understanding any single quantum algorithm.

What classical computers do in a hybrid system
Before QPU: Problem formulation (building the QUBO or Hamiltonian), parameter initialisation, circuit compilation, transpilation to native gates, embedding (for D-Wave), data preprocessing.

During QPU: For variational algorithms (QAOA, VQE), the classical optimiser runs continuously it reads expectation values from the QPU and computes new parameters. The QPU only evaluates the circuit; the classical machine decides what circuit to run next.

After QPU: Post-processing the raw measurement results, decoding the solution, constraint checking, majority voting over multiple runs, quality verification against classical bounds.
💡 The QPU is a subroutine, not a computer

Think of the QPU as a very specialised function call inside a classical program. Your Python code calls qpu.run(circuit, shots=1000) and waits for a result, exactly like calling an API. The classical computer orchestrates everything the QPU provides one specific capability (sampling from a quantum distribution) that the classical host cannot do efficiently. This mental model is more accurate than "quantum computer solving a problem."

03 How to decide what runs on the QPU

The most common mistake in hybrid design is putting too much on the QPU. The QPU is slow, noisy, and expensive per shot it should only do the part that classical computers cannot do efficiently.

TaskRun onWhy
Problem formulation (QUBO, Hamiltonian)Classical CPUDeterministic, fast, no quantum advantage
Parameter optimisation (COBYLA, Adam)Classical CPUClassical optimisers are mature and fast

Circuit/cost function evaluation ⟨H⟩QPUThis is what the QPU is for sample from quantum distribution

⚠ QPU queue time dominates wall-clock time

On IBM Quantum, typical queue time is 5–30 minutes per job regardless of circuit size. A QAOA optimisation that needs 500 circuit evaluations, with 5 minutes per job, takes 40+ hours of wall-clock time even if each circuit runs in microseconds. This is why training VQCs and QAOAs on QPUs is impractical you run on simulator, then deploy to QPU only for the final evaluation. Never design a hybrid system where the classical optimiser waits for each QPU result one at a time.

04 Batching the most important QPU performance pattern

The single most effective optimisation for any hybrid system is batching multiple circuits into one QPU job. Instead of submitting one circuit per optimisation step and waiting in the queue each time, submit all the circuits you need in one batch.

1
Naive approach one circuit per QPU call
500 QAOA evaluations × 5 min queue each = 41 hours total. The circuit itself runs in microseconds. 99.9% of time is queue overhead. This is what most tutorials do it is impractical at any real scale.
2
Batched approach many circuits per QPU call
Transpile all 500 circuits at once. Submit as a single job with 500 circuits. One queue wait (5 min). All 500 circuits execute. Total: 5 minutes instead of 41 hours. In Qiskit: sampler.run([circuit_list], shots=1000) accepts a list of circuits in one call.
3
Primitive-based batching (Qiskit 1.x)
Qiskit's Estimator and Sampler primitives handle batching automatically. Pass a list of (circuit, observable) pairs to Estimator.run() and get all expectation values back in one QPU call. This is the correct way to run QAOA on real IBM hardware today.

05 Error mitigation getting better results from noisy QPUs

NISQ hardware is noisy. Every gate has an error rate of 0.1–1%. A circuit with 100 gates will have roughly 10–100 errors per shot. Raw QPU results are therefore unreliable you need error mitigation to extract useful signal.

TechniqueHow it worksCostWhen to use
Readout error mitigationMeasure |0⟩ and |1⟩ states to characterise measurement errors. Apply inverse calibration matrix to raw counts.2ⁿ calibration circuits (exponential limit to ≤15 qubits)Always measurement errors are the most common noise source
Zero-noise extrapolation (ZNE)Run circuit at multiple noise levels (scale gates), extrapolate to zero noise. Works without knowing the noise model.3–5× more circuit evaluationsWhen you need accurate expectation values for QAOA/VQE
Probabilistic error cancellation (PEC)Represent ideal circuit as linear combination of noisy circuits. Statistically cancel noise. Near-exact mitigation.Exponential in circuit error rate only viable for low-noise circuitsResearch use only too expensive for most practical circuits
Dynamical decouplingInsert carefully timed X gate pairs during idle qubit periods to cancel environmental noise.Minimal just extra gates during idle timeWhen qubits sit idle for many cycles (common in deep circuits)
✓ What Qiskit Runtime provides automatically

When you use Qiskit's Estimator primitive with resilience_level=1, it automatically applies readout error mitigation and a basic level of ZNE. This is the recommended starting point for any real QPU execution you get significantly better expectation values with no extra code. resilience_level=2 adds more aggressive mitigation at higher cost.

06 A complete hybrid architecture for production

Here is what a production hybrid quantum-classical system looks like the components, responsibilities, and interfaces that a real application needs.

Production hybrid system component responsibilities
Problem layer (classical): Takes the user's problem description and produces the formal representation (QUBO, Hamiltonian, circuit template). All domain logic lives here. Language: Python with domain libraries (NetworkX, cvxpy, OR-Tools).

Quantum layer (hybrid): Implements the quantum algorithm QAOA loop, VQE, annealing. Manages parameter optimisation, circuit construction, batching, and result aggregation. Uses Qiskit Runtime, Ocean SDK, or PennyLane.

QPU interface (quantum hardware): IBM Quantum, D-Wave Leap, IonQ, or a local simulator. The quantum layer treats this as a black box call with circuits/QUBO, receive measurements/samples.

Post-processing layer (classical): Decodes raw QPU output into problem solutions, checks constraint satisfaction, compares with classical baseline, formats result for the user.

Classical fallback: Every production hybrid system needs a classical solver that runs in parallel or as a fallback. If the QPU result is worse than the classical result, use the classical one. Honest hybrid systems always present both.

Build a 6-node weighted MaxCut graph and compute the exact optimal cut using brute-force enumeration. This is the classical baseline.

import numpy as np import networkx as nx from qiskit import QuantumCircuit from qiskit.circuit import Parameter from qiskit_aer import AerSimulator from qiskit.compiler import transpile from scipy.optimize import minimize import time # ── Layer 1: Problem definition ─────────────────────────────── G = nx.Graph() G.add_weighted_edges_from([ (0,1,1.0), (1,2,0.8), (2,3,1.2), (3,4,0.9), (4,5,1.1), (5,0,0.7), (0,3,0.5), (1,4,0.6), (2,5,0.4), ]) n = G.number_of_nodes() # Classical baseline: brute force def cut_value(x, G): return sum(d['weight'] for u,v,d in G.edges(data=True) if x[u]!=x[v]) t0 = time.time() best_classical = 0 best_x_classical = None for i in range(2**n): x = [(i >> k) & 1 for k in range(n)] cv = cut_value(x, G) if cv > best_classical: best_classical, best_x_classical = cv, x[:] t_classical = time.time() - t0 print(f"Classical optimal cut : {best_classical:.2f}") print(f"Classical partition : {best_x_classical}") print(f"Classical solve time : {t_classical*1000:.2f} ms")

Key patterns in this implementation
simulator.run(circuits, shots=SHOTS)Passing a list of circuits submits all of them in ONE QPU call. The result object has .get_counts(index) to retrieve results for each circuit. This is the batching pattern 200 evaluations in 11 QPU calls instead of 200.
result_cache[(g,b)]Caching avoids re-evaluating the same parameters if COBYLA revisits them. In a real QPU scenario this saves expensive queue time a production system should always cache evaluated parameter sets.
flush_batch()Submits any buffered circuits when the batch is full or when a result is needed immediately. This is the batching control logic the optimiser doesn't know about batching, only the cost function does.
